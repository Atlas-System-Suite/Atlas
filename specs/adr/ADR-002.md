# ADR-002: Capability and Protocol Separation

Status: Accepted

Date: 2026-06-28

---

## Context

Atlas uses a Provider-based architecture where Modules consume abstract services and Providers implement them.

The relationship between the abstract service identity and the concrete code contract needs to be formally defined.

Two approaches were considered: merging Capabilities and Interfaces into one concept, or keeping them separate.

---

## Decision

Capabilities and Protocols are kept as separate concepts with explicit definitions:

* **Capability** — A runtime service identity. Discovered, resolved, and injected by the Capability Registry. Example: "storage", "ai", "notifications".

* **Protocol** — The code contract (Python `typing.Protocol`) that implementations must satisfy. Example: `StorageProtocol`, `AIProtocol`.

* **Provider** — A concrete implementation of one or more Protocols, registered for one or more Capabilities. Example: `SQLiteProvider` implements `StorageProtocol` and registers for the "storage" Capability.

### Relationship

```
Capability (identity) → references → Protocol (contract)
Provider (implementation) → satisfies → Protocol (contract)
Provider (implementation) → registers for → Capability (identity)
```

---

## Rationale

### Why Separate

* **A single Capability can have multiple Protocol versions.** As the Storage Protocol evolves from v0.1 to v0.2, both versions can coexist under the same "storage" Capability. This enables backward compatibility.

* **Multiple Capabilities can reference the same Protocol.** A "primary-storage" and "backup-storage" Capability could both reference `StorageProtocol` but resolve to different Providers.

* **Capabilities carry runtime metadata** (availability, preferred provider, health status) that Protocols should not.

* **Protocols are static contracts** suitable for type checking. Capabilities are runtime constructs suitable for dependency injection.

---

## Alternatives Considered

### Merged Model

A Capability IS its Protocol. Simpler — one fewer abstraction layer.

Rejected because it conflates the code contract (static, compile-time) with the service identity (dynamic, runtime). This would make version negotiation and multi-provider resolution significantly harder.

### Service Locator Pattern

Components look up services by name/type from a central registry.

Rejected in favor of explicit Capability injection because service locators hide dependencies, making code harder to test and reason about.

---

## Consequences

* Two separate specification documents must be maintained: `Capabilities.md` and `Interface.md` (renamed to `Protocols.md` for clarity).
* The Capability Registry must store a mapping from Capability names to Protocol versions and Provider instances.
* Module manifests declare required Capabilities (not Protocols) — the Registry handles resolution.
* Protocol definitions live in `src/atlas/capabilities/` as Python Protocol classes.
* Type checkers verify Provider compliance against Protocols at development time.
* The Capability Registry verifies Provider compliance at runtime.
